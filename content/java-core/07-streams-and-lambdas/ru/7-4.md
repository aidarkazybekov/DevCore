## Title
Stream API

## Summary
The Stream API enables declarative, pipeline-based processing of collections. Streams are lazy, single-use, and consist of a source, zero or more intermediate operations, and a terminal operation that triggers computation.

## TL;DR
Stream-пайплайн: источник → ноль или более ленивых промежуточных операций → одна терминальная операция, запускающая выполнение. Стримы одноразовые, не хранят данные и позволяют short-circuit оптимизацию через терминалы вроде `findFirst` или `anyMatch`.

## Analogy
Stream-пайплайн — как сборочная линия. Сырые детали (элементы источника) едут по конвейеру; каждая станция (промежуточная операция) трансформирует или фильтрует их, не останавливая линию. Упаковочная станция в конце (терминальная операция) — единственная, которая реально отгружает товар: ничего не движется, пока эта станция не активируется. Ранний фильтр качества (`filter()`) означает, что меньше деталей проходит всю длину линии.

## What & Why
Stream API (Java 8) заменяет многословные императивные `for`-циклы на компонуемые, читаемые пайплайны. Лень — ключевое дизайн-решение: промежуточные операции создают рецепт, а не результат. `filter` перед `map` перед `findFirst()` прерывается после первого совпадения — полный обход не нужен. Это важно для больших наборов данных, источников, работающих через I/O, и производительно-критичных путей, где раннее завершение избегает лишней работы.

## How It Works
**Промежуточные операции** (`filter`, `map`, `flatMap`, `sorted`, `distinct`, `limit`, `skip`) возвращают новый `Stream`, описывающий следующий шаг. Stateless-операции (`filter`, `map`, `flatMap`, `peek`) обрабатывают каждый элемент независимо. Stateful (барьерные) операции (`sorted`, `distinct`) должны буферизировать все вышестоящие элементы перед выдачей — они нарушают поэлементный стриминг. **Терминальные операции** (`collect`, `forEach`, `reduce`, `count`, `findFirst`, `anyMatch`) запускают обход и закрывают стрим; после терминала стрим повторно использовать нельзя. Short-circuit терминалы (`findFirst`, `findAny`, `anyMatch`, `allMatch`, `noneMatch`) останавливаются, как только результат определён, мощно сочетаясь с lazy evaluation. `flatMap` обрабатывает вложенные структуры, объединяя под-стримы поэлементно (один-ко-многим). Примитивные стримы (`IntStream`, `LongStream`, `DoubleStream`) устраняют накладные расходы на boxing в числовых пайплайнах.

## Gotcha
Стримы **одноразовые**: вызов второй терминальной операции бросает `IllegalStateException`. `sorted()` является барьером — он принудительно материализует весь upstream (O(n log n) по времени, O(n) по памяти) даже в ленивом пайплайне. Размещение `sorted()` перед `filter()` — частая ошибка производительности: сортируются все n элементов, затем большинство отбрасывается. Правильный порядок: сначала `filter()` для уменьшения n, затем `sorted()`.

## Recap
- Промежуточные операции ленивы — ничего не выполняется до вызова терминала.
- Stateless-операции: `filter`, `map`, `flatMap`, `peek`. Барьерные (stateful): `sorted`, `distinct`.
- Short-circuit терминалы останавливаются рано: `findFirst`, `anyMatch`, `allMatch`, `noneMatch`.
- `flatMap` = трансформация один-ко-многим + выравнивание.
- Порядок важен: `filter` перед `sorted` эффективнее, чем `sorted` перед `filter`.
- Стримы одноразовые — использованный стрим бросает `IllegalStateException` при повторном применении.

## Deep Dive
## Что такое Stream API?

Возможность работать с данными в функциональном стиле, появились в Java 8. Упрощенная работа с коллекциями.

Виды операций:
- Промежуточные (lazy)
- Терминальные (eager)

Промежуточные возвращают тот же Stream<>, а терминальные возвращают результат определенного типа. Все промежуточные операции выполняются лениво и пока не будет вызвана терминальная операция никаких действий на самом деле не происходит.

Кроме универсальных объектов существуют особые виды стримов для работы с примитивами: IntStream, LongStream и DoubleStream.

### Способы создания стрима
- Arrays.asList().stream() -- из коллекции
- Stream.of() -- из набора значений
- Arrays.stream() -- из массива
- \"hello\".chars() -- из строки
- Stream.iterate() -- бесконечный
- Stream.generate() -- бесконечный

### Промежуточные и терминальные методы

Промежуточные: filter(), skip(), distinct(), map(), peek(), limit(), sorted(), mapToInt(), mapToDouble(), mapToLong(), flatMap(), flatMapToInt(), flatMapToDouble(), flatMapToLong()

Терминальные: findFirst(), findAny(), count(), anyMatch(), allMatch(), noneMatch(), min(), max(), forEach(), forEachOrdered(), toArray(), reduce()

### Различия между forEach() и forEachOrdered()
- forEach() -- при параллельном выполнении порядок не гарантируется
- forEachOrdered() -- гарантируется

## Code
```java
import java.util.*;
import java.util.stream.*;

public class StreamApiDemo {

    record Employee(String name, String dept, double salary) {}

    public static void main(String[] args) {
        List<Employee> employees = List.of(
            new Employee("Alice", "Engineering", 95000),
            new Employee("Bob", "Engineering", 87000),
            new Employee("Charlie", "Marketing", 72000),
            new Employee("Diana", "Marketing", 68000),
            new Employee("Eve", "Engineering", 110000)
        );

        // 1. Filter, map, collect pipeline
        List<String> highEarners = employees.stream()
            .filter(e -> e.salary() > 80000)
            .map(Employee::name)
            .sorted()
            .collect(Collectors.toList());
        System.out.println("High earners: " + highEarners);

        // 2. reduce -- total salary
        double totalSalary = employees.stream()
            .mapToDouble(Employee::salary)
            .reduce(0.0, Double::sum);
        System.out.println("Total salary: " + totalSalary);

        // 3. flatMap -- flatten nested lists
        List<List<String>> nested = List.of(
            List.of("a", "b"),
            List.of("c", "d"),
            List.of("e")
        );
        List<String> flat = nested.stream()
            .flatMap(Collection::stream)
            .collect(Collectors.toList());
        System.out.println("Flat: " + flat); // [a, b, c, d, e]

        // 4. Short-circuiting -- lazy evaluation
        Optional<Employee> first = employees.stream()
            .filter(e -> e.dept().equals("Marketing"))
            .findFirst();
        first.ifPresent(e -> System.out.println("First marketing: " + e.name()));

        boolean anyHighEarner = employees.stream()
            .anyMatch(e -> e.salary() > 100000);
        System.out.println("Any > 100k? " + anyHighEarner);

        // 5. Primitive streams to avoid boxing
        IntStream.rangeClosed(1, 10)
            .filter(n -> n % 2 == 0)
            .forEach(n -> System.out.print(n + " ")); // 2 4 6 8 10
        System.out.println();

        // 6. Stream.of, generate, iterate
        Stream.iterate(1, n -> n <= 100, n -> n * 2)
            .forEach(n -> System.out.print(n + " ")); // 1 2 4 8 16 32 64
        System.out.println();

        // 7. teeing collector (Java 12+) -- two reductions at once
        var stats = employees.stream()
            .collect(Collectors.teeing(
                Collectors.counting(),
                Collectors.averagingDouble(Employee::salary),
                (count, avg) -> "Count: " + count + ", Avg: " + avg
            ));
        System.out.println(stats);
    }
}
```

## Tip
Always place filter() before map() and sorted() in a pipeline. Filtering first reduces the number of elements that expensive downstream operations must process.

## Interview
### [7-4-q0 | junior] What is the difference between intermediate and terminal operations? Give examples of each.
Intermediate operations (filter, map, flatMap, sorted, distinct, limit) return a new Stream and are lazy -- they do nothing until a terminal operation is invoked. Terminal operations (collect, forEach, reduce, count, findFirst, anyMatch) trigger pipeline execution and consume the stream. After a terminal operation, the stream cannot be reused.

### [7-4-q1 | mid] Explain the difference between map() and flatMap() with a practical example.
map() applies a one-to-one transformation: each element produces exactly one output. flatMap() applies a one-to-many transformation and flattens the result: each element produces zero or more elements that are merged into a single stream. Example: if you have List<Order> where each Order has List<LineItem>, orders.stream().map(Order::getItems) gives Stream<List<LineItem>>, while orders.stream().flatMap(o -> o.getItems().stream()) gives Stream<LineItem> -- a flat stream of all line items across all orders.

### [7-4-q2 | senior] How does stream laziness interact with stateful intermediate operations like sorted()? What are the performance implications?
Laziness means elements are processed one-at-a-time through the pipeline. However, sorted() is a barrier operation that must consume the entire upstream before emitting any elements, forcing full materialization into memory. This breaks the pipeline's streaming nature and has O(n log n) time and O(n) memory cost. If sorted() precedes limit(k), all elements are still sorted even though only k are needed. The optimization strategy is to push filters before sorted() to reduce the dataset, use min()/max() instead of sorted().findFirst(), or consider external sorting for very large datasets that exceed memory.

## Checkpoint
### [7-4-cp0] В чём разница между stateless и stateful промежуточными операциями? Приведите по одному примеру каждой.
Stateless промежуточная операция обрабатывает каждый элемент независимо, не нуждаясь в других элементах — например, `filter(e -> e.salary() > 80_000)` или `map(Employee::name)`. Stateful промежуточная операция должна накапливать элементы перед выдачей результата — например, `sorted()` обязан увидеть весь upstream перед выдачей первого отсортированного элемента. Stateful-операции нарушают поэлементный стриминг, увеличивают расход памяти и препятствуют short-circuit оптимизации ниже по пайплайну.

### [7-4-cp1] Почему размещение `sorted()` перед `limit(5)` снижает производительность и как это исправить?
`sorted()` является барьером, который принудительно материализует весь upstream и сортирует все n элементов (O(n log n) по времени, O(n) по памяти) перед передачей чего-либо в `limit(5)`. Хотя нужно только 5 элементов, обрабатываются все n. Исправление: разместите `filter()` перед `sorted()` для уменьшения набора данных, или замените `sorted().findFirst()` на `min(comparator)` / `max(comparator)` — им нужен только один проход O(n).

## Key Terms
### intermediate operation
Операция Stream, трансформирующая пайплайн и возвращающая новый `Stream`. Промежуточные операции ленивы — они не выполняются, пока не вызвана терминальная операция. Примеры: `filter`, `map`, `flatMap`, `sorted`, `distinct`, `limit`, `skip`.

### terminal operation
Операция Stream, запускающая выполнение пайплайна и производящая результат или побочный эффект. После терминальной операции стрим исчерпан и не может быть повторно использован. Примеры: `collect`, `forEach`, `reduce`, `count`, `findFirst`, `anyMatch`.

### lazy evaluation
Модель выполнения Stream API, при которой промежуточные операции создают рецепт вычисления, но не выполняют никакой работы до вызова терминальной операции. Lazy evaluation позволяет short-circuit оптимизацию: терминалы вроде `findFirst` или `anyMatch` прекращают обработку элементов, как только результат определён, избегая полного обхода источника.
