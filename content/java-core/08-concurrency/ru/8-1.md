## Title
Thread Lifecycle

## Summary
Java threads transition through NEW, RUNNABLE, BLOCKED, WAITING, TIMED_WAITING, and TERMINATED states. Understanding these states and the methods that trigger transitions is foundational for debugging concurrency issues and answering interview questions.

## TL;DR
Java-поток проходит через шесть состояний, управляемых вызовами методов и конкуренцией за монитор. Ключевое отличие: `Thread.sleep()` удерживает блокировки; `Object.wait()` — освобождает. Всегда используйте `while`-цикл вокруг `wait()`.

## Analogy
Представьте поток как рабочего в очереди задач. Только что обученный (NEW), он вступает в рабочую очередь (RUNNABLE) и ждёт своей очереди на CPU. Если нужная ему комната заперта кем-то другим (BLOCKED), он ждёт у двери. Если задача требует сигнала от коллеги (WAITING) или он ставит будильник на перерыв (TIMED_WAITING), он добровольно отступает в сторону. Когда работа завершена, он навсегда отмечается как выбывший (TERMINATED).

## What & Why
Java-приложения по своей природе конкурентны — JVM сама запускает несколько потоков для GC, обработки сигналов и финализаторов наряду с потоками приложения. Понимание шести состояний потока необходимо для диагностики дедлоков (потоки, застрявшие в BLOCKED), проблем liveness (потоки в WAITING без уведомителя) и корректной обработки прерываний. Различие между `sleep` и `wait` — одна из самых частых ловушек на интервью.

## How It Works
`new Thread(runnable)` создаёт поток в состоянии **NEW**. Вызов `start()` регистрирует его в планировщике ОС — поток становится **RUNNABLE** (доступен для CPU; фактическое выполнение зависит от планировщика). Планировщик ОС делит процессорное время, выполняя **переключение контекста** каждый раз, когда заменяет один RUNNABLE-поток другим. При попытке войти в `synchronized`-блок, занятый другим потоком, поток переходит в **BLOCKED** до освобождения монитора. `Object.wait()` атомарно освобождает монитор и переводит поток в **WAITING**; вызов `notify()` или `notifyAll()` возвращает его в RUNNABLE после повторного захвата монитора. `Thread.sleep(ms)` переводит поток в **TIMED_WAITING**, не освобождая удерживаемые блокировки. Поток достигает **TERMINATED**, когда `run()` завершается или необработанное исключение распространяется вверх. `Thread.interrupt()` устанавливает флаг; если поток находится в WAITING или TIMED_WAITING, немедленно бросается `InterruptedException` и флаг сбрасывается — всегда восстанавливайте его через `Thread.currentThread().interrupt()`.

## Gotcha
`Thread.sleep()` **не освобождает** блокировки. Если спящий поток удерживает монитор, другие потоки, нуждающиеся в нём, остаются в BLOCKED на всё время сна. Используйте `Object.wait()` (внутри `synchronized`) для освобождения блокировки во время паузы. Спецификация JVM допускает spurious wakeups: поток в WAITING может проснуться без `notify()`. Всегда оборачивайте `wait()` в `while`-цикл с проверкой реального условия — никогда не `if`.

## Recap
- NEW → RUNNABLE через `start()`.
- RUNNABLE → BLOCKED при конкуренции за монитор (вход в `synchronized`).
- RUNNABLE → WAITING через `wait()` / `join()` / `park()` — монитор освобождается.
- RUNNABLE → TIMED_WAITING через `sleep(ms)` / `wait(ms)` — `sleep` удерживает блокировки.
- TERMINATED при завершении `run()` или необработанном исключении.
- `wait()` требует `synchronized`; оборачивайте в `while` для защиты от spurious wakeups.
- `InterruptedException`: всегда восстанавливайте флаг прерывания или пробрасывайте.

## Deep Dive
## В каких состояниях может находиться поток?

- **NEW** -- объект класса Thread создан, но еще не запущен.
- **RUNNABLE** -- готов к выполнению, но планировщик еще не выбрал его.
- **BLOCKED** -- заблокирован, ожидает освобождения монитора.
- **WAITING** -- ожидает уведомления (notify / notifyAll), чтобы продолжить выполнение.
- **TIMED_WAITING** -- такой же как WAITING, но с ограничением по времени.
- **TERMINATED** -- завершил выполнение и больше не активен.

## Чем отличается процесс от потока?

**Процесс** -- программа во время выполнения, независимый объект, которому выделены системные ресурсы. Каждый процесс выполняется в отдельном адресном пространстве.

**Поток** (thread) -- определенный способ выполнения процесса, минимальная единица выполнения программы. Потоки в одном процессе имеют общий доступ к памяти и ресурсам.

## Каким образом можно создать поток?

- Создать потомка класса Thread и переопределить метод run()
- Реализовать функциональный интерфейс Runnable::void run и передать в конструктор Thread
- Реализовать функциональный интерфейс Callable::V call throws Exception и передать в конструктор Thread

## Чем отличаются Thread и Runnable и Callable?

- Thread -- это класс, некоторая надстройка над физическим потоком.
- Runnable -- функциональный интерфейс имеющий один метод void run()
- Callable -- функциональный интерфейс имеющий один метод V call() throws Exception
- Runnable появился в Java 1.0, а Callable в Java 5.0 в пакете java.util.concurrent

## Чем отличаются run() и start()?

run() -- определяет работу для потока, но выполняет её в текущем потоке, а не в новом.
start() -- создает новый поток выполнения, вызывает системные ресурсы для запуска нового потока, а затем внутренне вызывает метод run().

## Что такое Thread.sleep()?

Статический метод класса Thread, который приостанавливает выполнение текущего потока на заданное количество миллисекунд. Во время выполнения метода поток может быть прерван другим потоком, что вызовет InterruptedException.

## Что такое InterruptedException?

InterruptedException возникает, когда поток в состоянии ожидания прерывается другим потоком с помощью метода interrupt().
- interrupt() -- устанавливает флаг прерывания потока
- interrupted() -- статический метод, позволяющий получить состояние флага и сбросить его
- isInterrupted() -- метод получения состояния флага прерывания

## Code
```java
public class ThreadLifecycleDemo {

    private static final Object lock = new Object();

    public static void main(String[] args) throws InterruptedException {
        // 1. Thread states demonstration
        Thread thread = new Thread(() -> {
            try {
                // RUNNABLE -> TIMED_WAITING
                Thread.sleep(100);

                // RUNNABLE -> WAITING (via wait)
                synchronized (lock) {
                    lock.wait(); // releases the lock, enters WAITING
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt(); // restore flag
                System.out.println("Thread was interrupted");
            }
        });

        System.out.println("State after new: " + thread.getState()); // NEW

        thread.start();
        System.out.println("State after start: " + thread.getState()); // RUNNABLE

        Thread.sleep(50);
        System.out.println("State during sleep: " + thread.getState()); // TIMED_WAITING

        Thread.sleep(100);
        System.out.println("State during wait: " + thread.getState()); // WAITING

        // Wake it up
        synchronized (lock) {
            lock.notify();
        }
        thread.join(); // wait for thread to finish
        System.out.println("State after join: " + thread.getState()); // TERMINATED

        // 2. Proper wait/notify pattern with condition loop
        Thread producer = new Thread(() -> {
            synchronized (lock) {
                System.out.println("Producer: setting condition and notifying");
                lock.notifyAll();
            }
        });

        Thread consumer = new Thread(() -> {
            synchronized (lock) {
                // Always use while loop, NOT if, to guard against spurious wakeups
                while (!Thread.currentThread().isInterrupted()) {
                    try {
                        lock.wait(); // releases lock, waits for notify
                        break;
                    } catch (InterruptedException e) {
                        Thread.currentThread().interrupt();
                    }
                }
                System.out.println("Consumer: woke up and proceeding");
            }
        });

        consumer.start();
        Thread.sleep(50); // ensure consumer is waiting
        producer.start();

        consumer.join();
        producer.join();

        // 3. Interrupt handling
        Thread worker = new Thread(() -> {
            while (!Thread.currentThread().isInterrupted()) {
                // do work
                try {
                    Thread.sleep(1000);
                } catch (InterruptedException e) {
                    System.out.println("Worker interrupted during sleep");
                    Thread.currentThread().interrupt(); // re-set flag
                    break; // exit gracefully
                }
            }
            System.out.println("Worker exiting cleanly");
        });
        worker.start();
        Thread.sleep(100);
        worker.interrupt(); // request cancellation
        worker.join();
    }
}
```

## Tip
When catching InterruptedException, always either re-throw it or restore the interrupt flag with Thread.currentThread().interrupt(). Swallowing the exception silently breaks the interruption contract and prevents proper shutdown.

Spring's @Async annotation causes methods to execute on a separate thread managed by a TaskExecutor (which wraps ExecutorService). ## Interview
### [8-1-q0 | junior] What are the six thread states in Java, and what causes transitions between them?
NEW: created but not started. RUNNABLE: after start(), eligible for CPU time. BLOCKED: waiting to acquire a synchronized lock. WAITING: called wait(), join(), or park() with no timeout. TIMED_WAITING: called sleep(ms), wait(ms), join(ms), or parkNanos(). TERMINATED: run() completed or uncaught exception. Key transitions: start() moves NEW to RUNNABLE; entering a contended synchronized block moves to BLOCKED; calling wait() moves to WAITING and releases the lock; sleep() moves to TIMED_WAITING but retains locks.

### [8-1-q1 | mid] Why should Object.wait() always be called in a while loop rather than an if statement?
Because of spurious wakeups: the JVM specification allows threads to wake from wait() without being notified. If wait() is in an if block, the thread proceeds even though the condition it was waiting for may not be true. A while loop re-checks the condition after every wakeup, ensuring the thread only proceeds when the actual business condition is met. Pattern: synchronized(lock) { while (!condition) { lock.wait(); } // proceed }.

### [8-1-q2 | senior] Compare platform threads to virtual threads (Project Loom). When would you still prefer platform threads?
Virtual threads are JVM-managed, lightweight (~kilobytes of stack), and can scale to millions of concurrent tasks. They are ideal for I/O-bound workloads where threads mostly wait (web servers, database calls). Platform threads map 1:1 to OS threads (~1MB stack) and are managed by the OS scheduler. Prefer platform threads for CPU-bound computation where you need predictable scheduling and OS-level thread affinity, when using ThreadLocal heavily (virtual threads can have millions of instances, wasting memory per-ThreadLocal), when pinned to native code via JNI/FFI, or when using synchronized blocks that would pin virtual threads to carrier threads (ReentrantLock is preferred with virtual threads).

## Checkpoint
### [8-1-cp0] Опишите разницу между потоком в состоянии BLOCKED и потоком в состоянии WAITING.
BLOCKED-поток конкурирует за монитор, удерживаемый другим потоком — он пассивно поставлен JVM в очередь ожидания `synchronized`-ресурса. WAITING-поток добровольно освободил свой монитор через `Object.wait()` и припаркован до явного уведомления через `notify()` / `notifyAll()` или прерывания. Ключевое различие: BLOCKED — невольная конкуренция; WAITING — намеренная передача управления в ожидании внешнего сигнала.

### [8-1-cp1] Почему `Object.wait()` нужно оборачивать в `while`-цикл, а не в `if`?
Спецификация JVM допускает spurious wakeups — поток в WAITING может проснуться без соответствующего вызова `notify()`. Блок `if` позволил бы потоку продолжить выполнение, даже если ожидаемое условие всё ещё ложно. `while`-цикл перепроверяет условие после каждого пробуждения: `synchronized(lock) { while (!condition) { lock.wait(); } }`.

## Key Terms
### thread
Наименьшая единица выполнения CPU внутри Java-процесса. Каждый поток имеет собственный JVM Stack и PC Register, но разделяет Heap и Metaspace со всеми остальными потоками в той же JVM. Потоки в состоянии RUNNABLE конкурируют за CPU-время, управляемое планировщиком ОС.

### monitor
Встроенная блокировка, связанная с каждым Java-объектом, используемая для обеспечения взаимного исключения через `synchronized`-блоки и методы. Поток должен владеть монитором объекта, чтобы вызвать на нём `wait()`, `notify()` или `notifyAll()`. Попытка без монитора бросает `IllegalMonitorStateException`.

### context switch
Действие операционной системы по сохранению состояния выполняющегося потока и восстановлению состояния другого для его выполнения. Context switch позволяет нескольким потокам разделять CPU-ядро, но несёт накладные расходы; высокое количество потоков или частые блокирующие операции увеличивают частоту переключений и могут снизить пропускную способность.
