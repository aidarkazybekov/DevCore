## Title
Thread Lifecycle

## Summary
Java threads transition through NEW, RUNNABLE, BLOCKED, WAITING, TIMED_WAITING, and TERMINATED states. Understanding these states and the methods that trigger transitions is foundational for debugging concurrency issues and answering interview questions.

## TL;DR
A Java thread moves through six states driven by method calls and monitor contention. The critical distinction: `Thread.sleep()` keeps locks; `Object.wait()` releases them. Always use a `while` loop around `wait()`.

## Analogy
Think of a thread as a worker in a task queue. Fresh out of training (NEW), they join the work queue (RUNNABLE) and wait their turn on the CPU. If a room they need is locked by someone else (BLOCKED), they wait at the door. If their task requires a signal from a colleague (WAITING) or they set a timed alarm for a break (TIMED_WAITING), they step aside voluntarily. When the work is done, they clock out permanently (TERMINATED).

## What & Why
Java applications are inherently concurrent — the JVM itself runs multiple threads for GC, signal handling, and finalizers, alongside application threads. Understanding the six thread states is essential for diagnosing deadlocks (threads perpetually stuck in BLOCKED), liveness failures (threads stuck in WAITING with no notifier), and interrupt handling. The `sleep` vs `wait` distinction is one of the most common interview gotchas.

## How It Works
`new Thread(runnable)` creates a thread in **NEW** state. Calling `start()` registers it with the OS scheduler — the thread becomes **RUNNABLE** (eligible for CPU time; actual execution depends on the scheduler). The OS scheduler shares CPU time by performing a **context switch** each time it swaps one RUNNABLE thread for another. When a thread tries to enter a `synchronized` block held by another thread, it moves to **BLOCKED** until the monitor is released. `Object.wait()` atomically releases the monitor and moves the thread to **WAITING**; a `notify()` or `notifyAll()` call moves it back to RUNNABLE after re-acquiring the monitor. `Thread.sleep(ms)` moves the thread to **TIMED_WAITING** without releasing any locks it holds. A thread reaches **TERMINATED** when `run()` returns or an uncaught exception propagates. `Thread.interrupt()` sets a flag; if the thread is in WAITING or TIMED_WAITING, it immediately throws `InterruptedException` and clears the flag — always restore it with `Thread.currentThread().interrupt()`.

## Gotcha
`Thread.sleep()` does **not** release locks. If a sleeping thread holds a monitor, other threads needing that monitor are stuck in BLOCKED for the full sleep duration. Use `Object.wait()` (inside `synchronized`) to release the lock during a pause. Spurious wakeups are allowed by the JVM spec: a thread in WAITING can wake without a `notify()`. Always wrap `wait()` in a `while` loop checking the actual condition — never an `if`.

## Recap
- NEW → RUNNABLE via `start()`.
- RUNNABLE → BLOCKED on monitor contention (entering `synchronized`).
- RUNNABLE → WAITING via `wait()` / `join()` / `park()` — monitor released.
- RUNNABLE → TIMED_WAITING via `sleep(ms)` / `wait(ms)` — `sleep` retains locks.
- TERMINATED when `run()` returns or throws uncaught.
- `wait()` requires `synchronized`; wrap in `while` for spurious wakeup safety.
- `InterruptedException`: always re-set the interrupt flag or re-throw.

## Deep Dive
A Java thread begins in the NEW state after construction (new Thread(runnable)) and transitions to RUNNABLE when start() is called. RUNNABLE encompasses both 'ready to run' and 'currently running' -- the OS scheduler decides which RUNNABLE threads actually execute on CPU cores. Calling run() directly does NOT start a new thread; it executes the Runnable on the caller's thread.

From RUNNABLE, a thread enters BLOCKED when it attempts to acquire a monitor lock held by another thread (entering a synchronized block or method). It enters WAITING when it calls Object.wait(), Thread.join(), or LockSupport.park() without a timeout. It enters TIMED_WAITING when it calls Thread.sleep(ms), Object.wait(ms), Thread.join(ms), or LockSupport.parkNanos(). The thread returns to RUNNABLE when the blocking condition resolves.

Thread.sleep() pauses the current thread but does NOT release any locks it holds -- this is a common source of deadlocks. Object.wait() DOES release the monitor lock and must be called within a synchronized block. The woken thread must re-acquire the lock before continuing, so wait() should always be called in a while loop checking the condition, not an if statement, to guard against spurious wakeups.

Thread creation is expensive: each thread allocates a ~1MB stack by default. For this reason, production code uses thread pools (ExecutorService) rather than raw threads. Java 21 introduced virtual threads (Project Loom) which are lightweight, JVM-managed threads with minimal stack footprint that can scale to millions of concurrent tasks.

Thread.interrupt() sets the thread's interrupt flag. If the thread is in WAITING or TIMED_WAITING, it throws InterruptedException and clears the flag. If the thread is RUNNABLE, the flag is simply set and must be checked with Thread.interrupted() or isInterrupted(). Proper interrupt handling -- catching InterruptedException, restoring the interrupt flag, and terminating gracefully -- is essential for well-behaved concurrent code.

## Code
```java
public class ThreadLifecycleDemo {

    private static final Object lock = new Object();

    public static void main(String[] args) throws InterruptedException {
        // 1. Thread states demonstration
        Thread thread = new Thread(() -> {
            try {
                // RUNNABLE -> TIMED_WAITING
                Thread.sleep(100);

                // RUNNABLE -> WAITING (via wait)
                synchronized (lock) {
                    lock.wait(); // releases the lock, enters WAITING
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt(); // restore flag
                System.out.println("Thread was interrupted");
            }
        });

        System.out.println("State after new: " + thread.getState()); // NEW

        thread.start();
        System.out.println("State after start: " + thread.getState()); // RUNNABLE

        Thread.sleep(50);
        System.out.println("State during sleep: " + thread.getState()); // TIMED_WAITING

        Thread.sleep(100);
        System.out.println("State during wait: " + thread.getState()); // WAITING

        // Wake it up
        synchronized (lock) {
            lock.notify();
        }
        thread.join(); // wait for thread to finish
        System.out.println("State after join: " + thread.getState()); // TERMINATED

        // 2. Proper wait/notify pattern with condition loop
        Thread producer = new Thread(() -> {
            synchronized (lock) {
                System.out.println("Producer: setting condition and notifying");
                lock.notifyAll();
            }
        });

        Thread consumer = new Thread(() -> {
            synchronized (lock) {
                // Always use while loop, NOT if, to guard against spurious wakeups
                while (!Thread.currentThread().isInterrupted()) {
                    try {
                        lock.wait(); // releases lock, waits for notify
                        break;
                    } catch (InterruptedException e) {
                        Thread.currentThread().interrupt();
                    }
                }
                System.out.println("Consumer: woke up and proceeding");
            }
        });

        consumer.start();
        Thread.sleep(50); // ensure consumer is waiting
        producer.start();

        consumer.join();
        producer.join();

        // 3. Interrupt handling
        Thread worker = new Thread(() -> {
            while (!Thread.currentThread().isInterrupted()) {
                // do work
                try {
                    Thread.sleep(1000);
                } catch (InterruptedException e) {
                    System.out.println("Worker interrupted during sleep");
                    Thread.currentThread().interrupt(); // re-set flag
                    break; // exit gracefully
                }
            }
            System.out.println("Worker exiting cleanly");
        });
        worker.start();
        Thread.sleep(100);
        worker.interrupt(); // request cancellation
        worker.join();
    }
}
```

## Tip
When catching InterruptedException, always either re-throw it or restore the interrupt flag with Thread.currentThread().interrupt(). Swallowing the exception silently breaks the interruption contract and prevents proper shutdown.

## Interview
### [8-1-q0 | junior] What are the six thread states in Java, and what causes transitions between them?
NEW: created but not started. RUNNABLE: after start(), eligible for CPU time. BLOCKED: waiting to acquire a synchronized lock. WAITING: called wait(), join(), or park() with no timeout. TIMED_WAITING: called sleep(ms), wait(ms), join(ms), or parkNanos(). TERMINATED: run() completed or uncaught exception. Key transitions: start() moves NEW to RUNNABLE; entering a contended synchronized block moves to BLOCKED; calling wait() moves to WAITING and releases the lock; sleep() moves to TIMED_WAITING but retains locks.

### [8-1-q1 | mid] Why should Object.wait() always be called in a while loop rather than an if statement?
Because of spurious wakeups: the JVM specification allows threads to wake from wait() without being notified. If wait() is in an if block, the thread proceeds even though the condition it was waiting for may not be true. A while loop re-checks the condition after every wakeup, ensuring the thread only proceeds when the actual business condition is met. Pattern: synchronized(lock) { while (!condition) { lock.wait(); } // proceed }.

### [8-1-q2 | senior] Compare platform threads to virtual threads (Project Loom). When would you still prefer platform threads?
Virtual threads are JVM-managed, lightweight (~kilobytes of stack), and can scale to millions of concurrent tasks. They are ideal for I/O-bound workloads where threads mostly wait (web servers, database calls). Platform threads map 1:1 to OS threads (~1MB stack) and are managed by the OS scheduler. Prefer platform threads for CPU-bound computation where you need predictable scheduling and OS-level thread affinity, when using ThreadLocal heavily (virtual threads can have millions of instances, wasting memory per-ThreadLocal), when pinned to native code via JNI/FFI, or when using synchronized blocks that would pin virtual threads to carrier threads (ReentrantLock is preferred with virtual threads).

## Checkpoint
### [8-1-cp0] Describe the difference between a thread in BLOCKED state and a thread in WAITING state.
A BLOCKED thread is competing for a monitor lock held by another thread — it is passively queued by the JVM waiting for the `synchronized` resource to become available. A WAITING thread has voluntarily given up its monitor via `Object.wait()` and is parked until explicitly notified via `notify()` or `notifyAll()`, or interrupted. The key difference: BLOCKED is involuntary contention; WAITING is a deliberate release of control pending an external signal.

### [8-1-cp1] Why must `Object.wait()` be wrapped in a `while` loop rather than an `if`?
The JVM specification permits spurious wakeups — a thread in WAITING can wake without a corresponding `notify()` call. An `if` block would allow the thread to proceed even though the condition it was waiting for is still false. A `while` loop re-checks the condition after every wakeup, ensuring the thread only continues when the actual predicate holds: `synchronized(lock) { while (!condition) { lock.wait(); } }`.

## Key Terms
### thread
The smallest unit of CPU execution within a Java process. Each thread has its own JVM Stack and PC Register but shares the Heap and Metaspace with all other threads in the same JVM. Threads in RUNNABLE state compete for CPU time managed by the OS scheduler.

### monitor
An intrinsic lock associated with every Java object, used to enforce mutual exclusion via `synchronized` blocks and methods. A thread must own an object's monitor to call `wait()`, `notify()`, or `notifyAll()` on that object. Attempting to do so without the monitor throws `IllegalMonitorStateException`.

### context switch
The operating system's action of saving the state of a running thread and restoring the state of another so it can execute. Context switches allow multiple threads to share a CPU core but incur overhead; high thread counts or frequent blocking operations increase context switch frequency and can degrade throughput.
