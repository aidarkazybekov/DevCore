## Title
JVM Architecture

## Summary
The JVM is the runtime engine that executes Java bytecode. Understanding its internal architecture — class loading, memory areas, and the execution engine — is foundational to debugging performance issues, memory leaks, and understanding why Java behaves the way it does in production.

## TL;DR
The JVM loads bytecode into distinct memory regions (Heap, Metaspace, JVM Stack) and executes it first via an interpreter, then compiles hot paths to native code via JIT (C1 → C2). `-Xmx` controls only the heap — total process memory is 1.5–2× larger.

## Analogy
Think of the JVM as a factory. The loading dock (Class Loader) receives raw materials (`.class` files) and stores blueprints in the warehouse (Metaspace). Workers (threads) each have their own personal workbench (JVM Stack). Finished goods accumulate in the shared storage hall (Heap). The foreman (Execution Engine) starts by reading instructions manually (interpreter) but, once a task repeats enough times, automates it with a machine (JIT compiler).

## What & Why
Java's "write once, run anywhere" promise depends entirely on the JVM abstraction: the Java compiler produces platform-neutral bytecode; the JVM translates that bytecode to native instructions at runtime. Because the heap, Metaspace, and thread stacks are separate budgets, developers who only watch `-Xmx` are surprised when `top` shows RSS far exceeding their configured heap. Every Java engineer must understand the memory layout to diagnose `OutOfMemoryError`, tune GC, and write thread-safe code.

## How It Works
The **Class Loader** reads `.class` files and places class metadata into Metaspace (native memory outside the heap). The **Runtime Data Areas** divide memory into two categories: shared areas (Heap for all object instances; Metaspace for class metadata and bytecode; static field **values** live in the Heap, inside each class's `java.lang.Class` object) and per-thread areas (JVM Stack of method frames, PC Register for the current instruction address, Native Method Stack for JNI calls). The **Execution Engine** starts as an interpreter and promotes hot methods through JIT tiers: C1 (Tiers 1–3, fast compilation) then C2 (Tier 4, aggressive optimizations — inlining, escape analysis, loop unrolling). OSR (On-Stack Replacement) lets the JVM compile a method while it is already executing. If an optimistic assumption is later invalidated (e.g. a new subclass appears), the JVM deoptimizes and falls back to the interpreter.

## Gotcha
`-Xmx` controls **only** the heap. Real process RSS also includes Metaspace (unbounded unless `-XX:MaxMetaspaceSize` is set), JIT CodeCache (default 240 MB), thread stacks (N threads × `-Xss`), direct buffers, and GC bookkeeping. Expect RSS ≈ 1.5–2× your `-Xmx`. Also: a `static` field's value lives in the Heap (in the `java.lang.Class` object), shared across all threads, so it is not thread-safe just because it is static.

## Recap
- **Class Loader**: `.class` → metadata in Metaspace (load → link → initialize).
- **Heap** (shared): all object instances and arrays, bounded by `-Xmx`.
- **Metaspace** (shared, native memory): class metadata + bytecode. Static field values live in the Heap (`java.lang.Class` object).
- **JVM Stack** (per-thread): frames holding local variables and the operand stack.
- **Execution Engine**: interpreter → C1 JIT → C2 JIT; warm-up takes seconds to minutes.
- Always set `-XX:MaxMetaspaceSize` in production.

## Deep Dive
## Three JVM subsystems

The JVM is an abstract computing machine that executes Java bytecode. It has three subsystems:

- **Class Loader Subsystem** — loading (`.class` → metadata in Metaspace), linking (verification → preparation of `static` fields → resolution of symbolic refs), and initialization (`<clinit>`).
- **Runtime Data Areas** — memory regions used while the program runs.
- **Execution Engine** — interpreter + JIT + garbage collector.

## Runtime Data Areas

**Shared across all threads:**

- **Heap** — all object instances and arrays. Split into Young Generation (Eden + S0 + S1) and Old Generation.
- **Method Area (Metaspace since Java 8)** — class metadata, Runtime Constant Pool, method bytecode. Lives in native memory, not bounded by `-Xmx`. Static field *values* live in the Heap (in the `java.lang.Class` object).

**Per-thread (each thread has its own):**

- **JVM Stack** — frames for each method call (local variables + operand stack + return address).
- **PC Register** — address of the current bytecode instruction.
- **Native Method Stack** — for JNI calls into native code.

> [!gotcha]
> A local reference variable lives on the stack, but the object it points to is on the heap. That's also why a `static` field is NOT automatically thread-safe — its value lives in the Heap (in the `java.lang.Class` object, shared), and every thread writes to the same slot.

## Execution Engine

- **Interpreter** — executes bytecode instruction-by-instruction. Instant start, slow steady state.
- **JIT Compiler (HotSpot)** — tracks hot methods and compiles them to native code. Tiered compilation: C1 (fast, Tier 1–3) → C2 (aggressive: inlining, escape analysis, loop unrolling, Tier 4).
- **OSR (On-Stack Replacement)** — can compile a method *while it's still running* (important for long loops in `main`).
- **Deoptimization** — if an optimistic assumption is invalidated (e.g. a new subclass appears), JVM falls back to the interpreter.

Tiered compilation is why Java apps need to warm up: the first few requests are cold and slow.

## Memory areas people forget

`-Xmx` only bounds the heap. Real process RSS is the sum of:

- Heap (`-Xmx`)
- Metaspace (`-XX:MaxMetaspaceSize` — always set this in prod)
- CodeCache for JIT-compiled code (`-XX:ReservedCodeCacheSize`, default 240MB)
- Direct buffers (`-XX:MaxDirectMemorySize` — used by NIO, Netty)
- Thread stacks (native memory: N threads × `-Xss`)
- GC bookkeeping, card tables, symbol tables

That's why process RSS is typically 1.5–2× the `-Xmx` you configured.

> [!production]
> Minimum prod flags: `-Xmx`, `-XX:MaxMetaspaceSize`, `-XX:+HeapDumpOnOutOfMemoryError`, `-XX:HeapDumpPath`. For native-memory debugging: `-XX:NativeMemoryTracking=summary` + `jcmd <pid> VM.native_memory summary`.

## Code
```java
// Exploring JVM memory areas at runtime
public class JvmArchitectureDemo {
    // Static field value -> stored in Heap (inside the java.lang.Class object)
    private static final String APP_NAME = "JVM Demo";

    // Instance field -> stored on the Heap (as part of the object)
    private int[] largeArray = new int[1_000_000];

    public static void main(String[] args) {
        // 'demo' reference is on the stack; the object is on the heap
        JvmArchitectureDemo demo = new JvmArchitectureDemo();

        // Runtime memory inspection
        Runtime rt = Runtime.getRuntime();
        System.out.println("Max heap:   " + rt.maxMemory() / 1024 / 1024 + " MB");
        System.out.println("Total heap: " + rt.totalMemory() / 1024 / 1024 + " MB");
        System.out.println("Free heap:  " + rt.freeMemory() / 1024 / 1024 + " MB");
        System.out.println("Used heap:  " +
            (rt.totalMemory() - rt.freeMemory()) / 1024 / 1024 + " MB");

        // Thread stack info
        System.out.println("\nActive threads: " + Thread.activeCount());
        System.out.println("Current thread stack depth: " +
            Thread.currentThread().getStackTrace().length);

        // Demonstrate StackOverflowError
        try {
            infiniteRecursion(0);
        } catch (StackOverflowError e) {
            System.out.println("\nStackOverflowError caught!");
            System.out.println("Stack frames before overflow vary by JVM and -Xss setting");
        }
    }

    // Each call adds a frame to the thread's JVM Stack
    private static void infiniteRecursion(int depth) {
        infiniteRecursion(depth + 1);
    }
}
```

## Tip
When you hear 'where does X live in memory?', think: static/class metadata -> Metaspace, objects -> Heap, local variable references -> Stack (but the object they point to is still on the heap).

## Spring
### concept
JVM Memory Areas

### springFeature
Spring Application Context & Bean Scopes

### explanation
Spring singleton beans live as objects on the heap for the entire application lifetime, referenced by the `ApplicationContext`. Prototype-scoped beans are also on the heap but become eligible for GC once your code releases the reference.

Why this matters for diagnosing Spring Boot issues:
- A large number of proxy classes (from AOP, CGLIB) bloats Metaspace — each proxy is a separately generated class.
- Too many singleton beans holding large state will exhaust the heap.
- `@Cacheable` without an eviction policy is a classic leak — the map keeps growing, referenced from a singleton service, GC can never reclaim.
- An `ApplicationContext` that isn't shut down cleanly on reload (common with embedded Tomcat + hot-reload) prevents classloader unload → Metaspace leak.

## Interview
### [1-1-q0 | junior] What are the main runtime data areas in the JVM, and which are shared between threads?
The JVM has five runtime data areas.

Shared across threads:
• Heap — all object instances live here (`new Foo()` → Heap). Split into Young Generation (Eden + S0 + S1) and Old Generation for GC.
• Method Area / Metaspace — class metadata, Runtime Constant Pool, method bytecode. Static field *values* live in the Heap (in the `java.lang.Class` object).

Per-thread (each thread owns its own):
• JVM Stack — frames for each method call (local variables + operand stack + return address).
• PC Register — address of the current bytecode instruction.
• Native Method Stack — used for JNI calls into native code.

Why it matters: shared areas need synchronization (two threads writing the same heap object → race condition), per-thread areas are safe by construction. That's why local primitives are inherently thread-safe, but an object's fields on the heap are not.

Common trap: `static` fields are NOT automatically thread-safe just because they're static — their values live in the Heap (in the `java.lang.Class` object, shared), so all threads touch the same slot.

### [1-1-q1 | junior] Where in JVM memory do these live: a primitive local variable, an object, a reference to that object, a `static` field, and a string literal? Walk through a concrete example.
Consider this code:

```java
class Order {
  static int counter = 0;        // (1)
  String code = "A-001";         // (2)
  public void process() {
    int qty = 5;                 // (3)
    Order other = new Order();   // (4)(5)
  }
}
```

(1) `counter` — static field → Heap (in the `java.lang.Class` object; one slot per JVM).
(2) `code` — instance field → lives inside the Order object on the Heap.
(3) `qty` — local primitive → JVM Stack of the current thread, inside the `process()` frame.
(4) `other` — local reference → JVM Stack (a 4/8-byte pointer, not the object).
(5) `new Order()` — the object itself → Heap (Eden space on first allocation).

The string literal `"A-001"` → String Pool. In Java 7+ the pool lives in the Heap; Metaspace only holds class names and metadata.

Key insight: the reference and the object are two different memory locations. When the method returns, the stack frame (with `qty` and `other`) is reclaimed automatically, but the Order instance on the Heap survives until no reachable references remain — then GC collects it.

### [1-1-q2 | mid] What's the difference between `StackOverflowError` and `OutOfMemoryError`? Can stack space cause an OOM? How do you diagnose this in production?
**StackOverflowError**: a thread exceeded its call stack depth — typically infinite/too-deep recursion. Stack size is controlled by `-Xss` (default ~512KB–1MB).

```java
void recurse() { recurse(); } // SOE after ~10-20K frames
```

**OutOfMemoryError** is an umbrella — several distinct failures share the name:
• `Java heap space` — `-Xmx` exhausted. Usually a memory leak (a collection grows unbounded).
• `GC overhead limit exceeded` — JVM spends >98% of time in GC and reclaims <2%.
• `Metaspace` — too many loaded classes (classloader leaks in app servers).
• `unable to create new native thread` — OS can't allocate stack space for a new thread. This IS an OOM caused by stack, but it's NOT a StackOverflow! Thousands of threads × 1MB stack each = gigabytes of native memory.
• `Direct buffer memory` — `-XX:MaxDirectMemorySize` exhausted (NIO, Netty).

Production diagnosis:
1. Always run with `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/var/logs/heap`.
2. Dump analysis: Eclipse MAT, VisualVM, JProfiler — use the Leak Suspects report.
3. Live inspection: `jcmd <pid> GC.heap_info`, `jstat -gc <pid> 1s`, `jmap -histo:live <pid>`.
4. For 'unable to create thread': `ulimit -a`, `cat /proc/<pid>/status | grep Threads`.

### [1-1-q3 | mid] What is JIT compilation? Why do Java apps need to 'warm up', and how does this affect SLAs and benchmarks?
The JVM Execution Engine runs in two modes:

**Interpreter** — reads bytecode instruction-by-instruction and executes it. Starts instantly, but slow (every loop iteration pays dispatch overhead).

**JIT Compiler** (HotSpot) — tracks 'hot' methods (invocation counter > threshold, default ~10K) and compiles them to native code. Tiered compilation:
• Tier 0: interpreter.
• Tier 1–3: C1 compiler (quick, simple compilation).
• Tier 4: C2 compiler (aggressive optimizations: inlining, escape analysis, loop unrolling).

Hence warm-up: the first few requests hit the interpreter — slow. As hot paths are compiled by C1 → C2, latency drops 10–100×.

Production consequences:
• For the first N seconds after startup the service is 'cold'. A Kubernetes readiness probe with a short timeout can kill the pod before warm-up finishes.
• Canary deploys: the new instance is slower than old ones — can blow your P99 latency.
• Microbenchmarks: NEVER measure performance without warm-up iterations. Use JMH — it handles warm-up for you.
• Mitigations: AOT compilation (GraalVM native-image), CDS (Class Data Sharing), synthetic traffic warm-up before adding to the load balancer.

Observability flags: `-XX:+PrintCompilation`, `-XX:+UnlockDiagnosticVMOptions -XX:+PrintInlining`.

### [1-1-q4 | senior] Why did Java 8 replace PermGen with Metaspace, and what are the production implications? Which flags must you always set?
**PermGen (pre-Java 8)** — a fixed region inside the Heap holding class metadata, interned strings, and statics. Problem: hard cap via `-XX:MaxPermSize` (default 64–82MB). In app servers (Tomcat, JBoss) with frequent hot deploys, classloaders often failed to unload → PermGen filled up → `OutOfMemoryError: PermGen space`. A classic bug: crash after ~10–20 redeploys.

**Metaspace (Java 8+)** — class metadata moved to native memory (outside the Heap). Key changes:
• By default, size is unbounded — grows until the process runs out of RAM.
• Interned strings moved to the regular Heap (dumpable, GC-able).
• Class metadata can be reclaimed when a classloader unloads.

Why this is dangerous in production:
• The old PermGen OOM failed loudly and early. A Metaspace leak now silently eats native memory until the Linux OOM killer takes down the whole node.
• Without a Metaspace cap, one classloader bug can consume all host memory and crash co-located services.

Mandatory production flags:
```
-XX:MaxMetaspaceSize=512m          # upper bound — must have
-XX:MetaspaceSize=256m             # initial size, avoids re-thresholds
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/logs/heap
```

Monitoring: `jcmd <pid> VM.metaspace` or `jstat -gcmetacapacity <pid>`. If Metaspace grows linearly without stabilizing → classloader leak. Common causes: ThreadLocals holding class references, JDBC driver deregistration failures, Spring contexts not shut down cleanly.

### [1-1-q5 | senior] A production service throws `OutOfMemoryError: Java heap space` about 6 hours after start. The heap grows slowly and monotonically. Walk through your step-by-step memory-leak diagnosis.
Playbook:

**1. Make sure a heap dump already exists.** The service must run with `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/var/logs/heap/`. If not, add it immediately and wait for the next crash (or capture a live dump before the crash: `jcmd <pid> GC.heap_dump /tmp/leak.hprof`).

**2. Confirm it's a leak, not normal growth.** Check GC logs (`-Xlog:gc*:file=gc.log` in Java 9+). Leak signals:
• After Full GC, used heap doesn't drop back to the previous baseline (monotonic growth).
• GC intervals shrink, durations grow.
• Old Gen peak after GC trends toward max heap.

**3. Analyze the dump in Eclipse MAT:**
• Leak Suspects Report — auto-identifies suspicious accumulators.
• Dominator Tree — which object retains the most heap.
• Histogram — top classes by count and size. Usual suspects: `HashMap$Node`, `char[]`/`byte[]`, `ArrayList`.
• Path to GC Roots (exclude weak/soft) for suspects — reveals who's holding the reference.

**4. Typical culprits:**
• A static collection used as a cache without eviction (Map keeps inserting, never removes).
• ThreadLocal in a thread pool — threads live forever, ThreadLocal values accumulate.
• Listeners/observers never unregistered.
• Resources not closed (Statement, ResultSet, InputStream).
• Non-static inner classes implicitly holding the enclosing instance.

**5. Quick live checks without a dump:**
```
jcmd <pid> GC.class_histogram | head -30   # top classes
jstat -gcutil <pid> 1s                     # live GC metrics
jmap -histo:live <pid> | head -30          # after a forced GC
```

**6. Fix + proof.** After the fix, run a multi-hour load test while watching Old Gen — the baseline must stabilize, not keep climbing.

### [1-1-q6 | senior] You set `-Xmx2g`, but `top` shows the Java process using 3.5GB RSS. Where did the extra 1.5GB go and how do you investigate?
`-Xmx` bounds ONLY the Java heap. Real process memory = heap + a pile of other regions, each with its own budget:

**What RSS is composed of:**
- **Heap** (`-Xmx`) — what you configured.
- **Metaspace** — class metadata in native memory. Without `-XX:MaxMetaspaceSize` it grows unbounded.
- **CodeCache** — JIT-compiled code. Default `-XX:ReservedCodeCacheSize=240m`. Can fill up in long-running services with large code paths.
- **Direct Memory** — `ByteBuffer.allocateDirect()`, Netty ByteBufs, some compression libs. Limit: `-XX:MaxDirectMemorySize` (defaults to `-Xmx`).
- **Thread stacks** — native memory: N threads × `-Xss` (default ~1MB). 1000 threads ≈ 1GB.
- **GC structures** — card tables, remembered sets (G1/ZGC), marking bitmaps.
- **Symbol Table, String Table** — native.
- **JNI allocations** — native libraries (JDBC drivers, libsnappy, jemalloc) can allocate outside JVM bookkeeping.
- **Glibc / malloc fragmentation** — allocator may hold onto 'free' blocks without returning them to the OS.

**Diagnosis:**

```
# 1. Enable Native Memory Tracking (requires restart)
-XX:NativeMemoryTracking=summary

# 2. Inspect the breakdown
jcmd <pid> VM.native_memory summary
jcmd <pid> VM.native_memory summary.diff   # delta since baseline

# 3. All region states
jcmd <pid> VM.info
jcmd <pid> VM.flags                        # all active flags
jstat -gcutil <pid>                        # GC utilization

# 4. Threads
jcmd <pid> Thread.print | grep -c '^"'    # thread count

# 5. Direct memory (via JMX)
jcmd <pid> GC.run                          # force a GC first
# then inspect java.nio:type=BufferPool over JMX
```

**Common culprits for 'invisible' memory:**
- Netty / Kafka client — direct buffers aren't bounded by `-Xmx`.
- JDBC pool with thousands of connections — each holds native resources.
- Wrong thread-pool sizing (Tomcat maxThreads=500+) — stacks silently eat GBs.
- Glibc arena fragmentation — fix with `MALLOC_ARENA_MAX=2` or switch to jemalloc.

**Rule of thumb for prod:** RSS budget ≈ `-Xmx` × 1.5–2×. If higher, dig in with NMT.

## Checkpoint
### [1-1-cp0] A static field is often assumed thread-safe. Where does it live, and why is that assumption wrong?
A `static` field's value lives in the Heap (in the `java.lang.Class` object), which is shared across all threads. Every thread in the JVM reads and writes the same memory slot simultaneously. Without explicit synchronization (`volatile`, `synchronized`, or an atomic type), concurrent writes create a race condition — the field is not thread-safe despite being `static`.

### [1-1-cp1] Your Java process shows 3.5 GB RSS with `-Xmx2g`. Name three memory regions outside the heap that account for the difference.
Metaspace (class metadata in native memory, unbounded by default without `-XX:MaxMetaspaceSize`), JIT CodeCache (`-XX:ReservedCodeCacheSize`, default 240 MB), and thread stacks (number of threads × `-Xss`, each typically ~1 MB). Direct buffers (`-XX:MaxDirectMemorySize`) and GC bookkeeping structures are additional contributors.

## Key Terms
### class loader
The JVM subsystem responsible for loading, linking, and initializing class files. It reads `.class` files from the classpath and places their metadata into Metaspace, making the class available for instantiation and execution.

### bytecode
Platform-neutral binary instructions stored in `.class` files, produced by the Java compiler. The JVM Execution Engine interprets or JIT-compiles bytecode into native machine instructions at runtime, enabling Java's "write once, run anywhere" portability.

### heap
The JVM runtime memory region shared by all threads where object instances and arrays are allocated. It is divided into Young Generation (Eden + Survivor spaces) and Old Generation, and its size is bounded by the `-Xmx` JVM flag.
