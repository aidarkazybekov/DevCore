## Title
HashMap Internals

## Summary
HashMap is the most frequently asked collection in Java interviews. It uses an array of buckets with hash-based indexing, handles collisions via linked lists that convert to red-black trees at threshold 8, and rehashes when the load factor (0.75) is exceeded.

## TL;DR
HashMap stores entries in a power-of-2 array of buckets, resolves collisions via linked lists that treeify at 8 nodes (O(n) → O(log n)), and doubles the table when entries exceed `capacity × 0.75`. Violating the `equals`/`hashCode` contract silently breaks lookups.

## Analogy
Imagine a library where each book is assigned a shelf number by hashing its ISBN. Most shelves hold one book. When a shelf overflows to 8 books, the librarian replaces the stack with a sorted filing cabinet (red-black tree) for faster lookup. When more than 75% of shelves are occupied, the library doubles in size and redistributes all books — each book moves at most one shelf over, determined by a single digit in its ISBN.

## What & Why
`HashMap` underpins `Map` usage everywhere in Java — configuration stores, caches, frequency counters. Its O(1) average-case `get`/`put` can degrade to O(n) under deliberate hash-collision attacks; the treeification safety net added in Java 8 caps worst-case at O(log n). Understanding the `equals`/`hashCode` contract is critical: a broken `hashCode` silently sends equal keys to different buckets, causing `get` to return `null` even though the key logically exists.

## How It Works
`put(key, value)` computes a spread hash (`hashCode()` XOR'd with its upper 16 bits) then indexes into the `Node<K,V>[]` table using `(capacity - 1) & hash` — a fast bitwise AND that works because capacity is always a power of 2. On a hash collision, it traverses or appends to the bucket's linked list. When that list reaches TREEIFY_THRESHOLD (8) and the table has at least MIN_TREEIFY_CAPACITY (64) buckets, the list converts to a red-black tree; it reverts below UNTREEIFY_THRESHOLD (6). When `size > capacity × load factor` (default 0.75), the table doubles: each entry either stays at its old index or moves to `old index + old capacity`, determined by a single bit — hashes are not recomputed. The `equals`/`hashCode` contract is mandatory: objects that are `equals()` must return the same `hashCode()`; violating this breaks bucket lookup.

## Gotcha
HashMap is **not thread-safe**. Concurrent modification corrupts internal state — in older Java versions, it could cause an infinite loop during resize. Always use `ConcurrentHashMap` in multithreaded code. Also, never mutate an object that is used as a HashMap key after insertion: changing the object changes its `hashCode`, so the entry moves to the wrong bucket and becomes unretrievable.

## Recap
- Default: 16 buckets, load factor 0.75, treeify at 8 nodes (table ≥ 64).
- Spread hash: `h ^ (h >>> 16)` distributes bits across the index range.
- Bucket index: `(n - 1) & hash` (fast, requires power-of-2 capacity).
- Treeify to red-black tree at 8; untreeify below 6.
- Rehash doubles capacity; entries move at most one slot using a single bit check.
- `equals`/`hashCode` contract: always override both together.
- Thread-safety: use `ConcurrentHashMap`.

## Deep Dive
HashMap stores entries in a Node<K,V>[] table (array of buckets). When you call put(key, value), it computes the bucket index as (n - 1) & hash(key), where n is the array length (always a power of 2) and hash() applies a spread function that XORs the upper 16 bits with the lower 16 bits of hashCode() to reduce collisions. If two keys map to the same bucket, they form a linked list (separate chaining).

Since Java 8, when a bucket's linked list reaches 8 nodes (TREEIFY_THRESHOLD), it converts to a balanced red-black tree, improving worst-case lookup from O(n) to O(log n). When the tree shrinks below 6 nodes (UNTREEIFY_THRESHOLD), it converts back to a linked list. Treeification only occurs if the table has at least 64 buckets (MIN_TREEIFY_CAPACITY); otherwise, the table is resized instead. This is a critical interview detail.

Resizing (rehashing) occurs when size exceeds capacity * loadFactor (default 0.75). The new capacity doubles, and every entry is redistributed. In Java 8+, rehashing is optimized: since capacity is always a power of 2, each entry either stays in the same bucket or moves to bucket (old index + old capacity), determined by a single bit check. This avoids recalculating all hashes.

The contract between equals() and hashCode() is fundamental: if two objects are equal via equals(), they MUST have the same hashCode(). Violating this contract causes HashMap to place equal keys in different buckets, making lookups fail silently. HashMap is not thread-safe; concurrent modification can cause infinite loops (in old Java versions) or corrupted data. Use ConcurrentHashMap for thread safety.

## Code
```java
import java.util.*;

public class HashMapInternals {
    public static void main(String[] args) {
        // Default: capacity=16, loadFactor=0.75
        Map<String, Integer> map = new HashMap<>();

        // Preallocate to avoid resizing
        // For 100 entries: 100 / 0.75 = 134, next power of 2 = 256
        Map<String, Integer> optimized = new HashMap<>(256);

        // How put() works internally:
        // 1. hash = spread(key.hashCode()) -- XOR upper/lower 16 bits
        // 2. index = (capacity - 1) & hash
        // 3. If bucket empty: insert new Node
        // 4. If bucket occupied: traverse chain
        //    - If key found (equals): replace value
        //    - If not found: append to chain
        // 5. If chain length >= 8 AND table >= 64: treeify
        // 6. If size > capacity * 0.75: resize (double)
        map.put("name", 1);
        map.put("name", 2); // replaces, returns old value 1

        // equals/hashCode contract demo
        Map<Person, String> people = new HashMap<>();
        Person p1 = new Person("Alice", 30);
        people.put(p1, "Engineer");
        System.out.println(people.get(new Person("Alice", 30))); // "Engineer"
        // Only works because Person overrides both equals AND hashCode
    }
}

class Person {
    private final String name;
    private final int age;

    Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Person)) return false;
        Person p = (Person) o;
        return age == p.age && Objects.equals(name, p.name);
    }

    @Override
    public int hashCode() {
        return Objects.hash(name, age); // MUST override with equals
    }
}
```

## Interview
### [6-2-q0 | junior] How does HashMap determine which bucket to place an entry in?
HashMap computes a hash by taking the key's hashCode() and XORing its upper 16 bits with its lower 16 bits (spread function) to reduce collisions. The bucket index is calculated as (capacity - 1) & hash, which works because capacity is always a power of 2. This is a fast bitwise AND that replaces the modulo operation.

### [6-2-q1 | mid] What happens when a HashMap bucket has too many collisions? What are the thresholds?
When a bucket's linked list reaches 8 nodes (TREEIFY_THRESHOLD), HashMap converts the list into a red-black tree, improving worst-case lookup from O(n) to O(log n). However, treeification only occurs if the table capacity is at least 64 (MIN_TREEIFY_CAPACITY); otherwise, it resizes the table instead. When removals shrink a tree below 6 nodes (UNTREEIFY_THRESHOLD), it converts back to a linked list. The gap between 8 and 6 prevents thrashing between list and tree forms.

### [6-2-q2 | senior] Explain the Java 8 resize optimization for HashMap. Why must capacity be a power of 2?
Power-of-2 capacity enables the bitwise AND trick: index = (capacity - 1) & hash, which is faster than modulo. During resize (doubling), each entry's new position depends on a single bit: the bit at the position of the old capacity. If that bit is 0, the entry stays at the same index; if 1, it moves to oldIndex + oldCapacity. This allows splitting each bucket into two using simple bit checks without recalculating hashes. Entries are split into 'lo' and 'hi' lists, maintaining order. This optimization, introduced in Java 8, also eliminated the possibility of infinite loops during concurrent resize that plagued Java 7's head-insertion rehashing.

## Checkpoint
### [6-2-cp0] You insert a custom object as a key, then call `get` with an equal object — it returns `null`. What is the most likely cause?
The class overrides `equals` but not `hashCode`. Two logically equal objects produce different hash codes, so they land in different buckets. The `get` call looks in the wrong bucket and finds nothing. Fix: always override `hashCode` whenever you override `equals` so that equal objects hash to the same bucket.

### [6-2-cp1] A bucket's linked list has 8 nodes. Under what condition does it NOT treeify, and what happens instead?
Treeification is skipped if the table capacity is below MIN_TREEIFY_CAPACITY (64). Instead of converting to a red-black tree, the entire table is resized (doubled), redistributing entries across more buckets and reducing chain length without the overhead of a tree structure.

## Key Terms
### load factor
A ratio (default 0.75) that controls when HashMap resizes. When the number of entries exceeds `capacity × load factor`, the table doubles. A lower load factor reduces collisions at the cost of more memory; a higher value saves memory but increases collision frequency.

### treeify
The process of converting a HashMap bucket's linked list into a red-black tree when the list reaches TREEIFY_THRESHOLD (8 nodes) and the table has at least MIN_TREEIFY_CAPACITY (64) buckets. Treeification improves worst-case lookup from O(n) to O(log n). The inverse — converting a tree back to a list — occurs below UNTREEIFY_THRESHOLD (6 nodes).

### hash collision
The condition in which two distinct keys produce the same bucket index after hashing. HashMap resolves collisions via separate chaining (a linked list or red-black tree per bucket). Frequent collisions degrade performance and can result from a poor `hashCode` implementation that distributes keys unevenly.
