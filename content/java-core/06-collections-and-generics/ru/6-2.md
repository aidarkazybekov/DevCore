## Title
HashMap Internals

## Summary
Внутри HashMap хранится массив (по умолчанию размер 16), где каждый элемент называется bucket. Каждый бакет может быть пустым, содержать связанный список или (в Java 8+) красно-черное дерево при 8 и более элементов. При превышении threshold 0.75 массив расширяется в два раза.

## TL;DR
HashMap хранит записи в массиве бакетов (степень двойки), разрешает коллизии через связанные списки, которые преобразуются в красно-чёрное дерево при 8 узлах (O(n) → O(log n)), и удваивает таблицу, когда записи превышают `capacity × 0.75`. Нарушение контракта `equals`/`hashCode` молча ломает поиск.

## Analogy
Представьте библиотеку, где каждой книге присваивается номер полки на основе хэша её ISBN. Большинство полок содержат одну книгу. Когда на полке скапливается 8 книг, библиотекарь заменяет стопку упорядоченным каталогом (красно-чёрным деревом) для быстрого поиска. Когда занято более 75% полок, библиотека удваивается в размере и перераспределяет все книги — каждая книга сдвигается максимум на одну полку, что определяется одной цифрой в её ISBN.

## What & Why
`HashMap` лежит в основе практически любого использования `Map` в Java — конфигурации, кэши, счётчики частот. Его среднее время O(1) для `get`/`put` может деградировать до O(n) при намеренных атаках с коллизиями; защитная сетка в виде преобразования в дерево (Java 8) ограничивает худший случай до O(log n). Знание контракта `equals`/`hashCode` критично: неправильный `hashCode` молча отправляет равные ключи в разные бакеты, и `get` возвращает `null`, хотя ключ логически существует.

## How It Works
`put(key, value)` вычисляет spread-хэш (`hashCode()` XOR с его старшими 16 битами), затем индексирует в массив `Node<K,V>[]` через `(capacity - 1) & hash` — быстрое побитовое AND, работающее потому, что capacity всегда является степенью двойки. При коллизии хэша метод обходит или добавляет в связанный список бакета. Когда список достигает TREEIFY_THRESHOLD (8) и таблица имеет не менее MIN_TREEIFY_CAPACITY (64) бакетов, список преобразуется в красно-чёрное дерево; обратное преобразование происходит ниже UNTREEIFY_THRESHOLD (6). Когда `size > capacity × load factor` (по умолчанию 0.75), таблица удваивается: каждая запись либо остаётся на старом индексе, либо перемещается на `old index + old capacity`, что определяется одним битом без пересчёта хэшей. Контракт `equals`/`hashCode` обязателен: объекты, равные через `equals()`, должны возвращать одинаковый `hashCode()`.

## Gotcha
HashMap **не является thread-safe**. Конкурентное изменение нарушает внутреннее состояние — в старых версиях Java это могло привести к бесконечному циклу при resize. Всегда используйте `ConcurrentHashMap` в многопоточном коде. Также никогда не изменяйте объект, используемый как ключ HashMap после вставки: изменение объекта меняет его `hashCode`, запись попадает в неверный бакет и становится недостижимой.

## Recap
- По умолчанию: 16 бакетов, load factor 0.75, treeify при 8 узлах (таблица ≥ 64).
- Spread-хэш: `h ^ (h >>> 16)` распределяет биты по диапазону индексов.
- Индекс бакета: `(n - 1) & hash` (быстро, требует capacity как степень двойки).
- Treeify в красно-чёрное дерево при 8; обратно при < 6.
- Rehash удваивает capacity; записи перемещаются максимум на один слот через проверку одного бита.
- Контракт `equals`/`hashCode`: всегда переопределяйте оба вместе.
- Thread-safety: используйте `ConcurrentHashMap`.

## Deep Dive
Внутри HashMap хранится массив (по умолчанию размер 16), где каждый элемент называется bucket («корзина»). Каждый бакет может быть пустым, содержать связанный список или в Java 8+ красно-черное дерево при 8 и более элементах. При превышении threshold 0.75 массив расширяется в два раза.

Бакет может быть связанным списком (Node) или красно-черным деревом (TreeNode) (своя реализация, TreeMap не используется).

Node содержит: Key, Value, Hash of key, nextNode.
TreeNode содержит: Parent, Left, Right, Prev, Boolean red.

Как работает вставка (put):
1. Вычисляется хэш ключа.
2. Определяется индекс бакета путем деления с остатком хэша на (длину массива - 1).
3. Если бакет пуст -- создаем Node и кладем в бакет.
4. Если бакет не пуст (коллизия):
   - Сравниваем хэши ключей элементов в бакете.
   - Если хэш ключей совпал, вызывается equals() ключа для проверки.
   - Если equals() == true -> перезаписываем на новое значение.
   - Если equals() == false -> продолжаем искать в этом бакете.
5. Проверяем длину цепочки (Java 8+): если длина >= 8, связанный список превращается в красно-черное дерево. Обратное превращение дерева в список происходит при количестве узлов <= 6.
6. Проверка необходимости расширения.

Почему ключ в HashMap не обязан реализовывать Comparable, но список удается преобразовать в красно-черное дерево? Алгоритм сравнения ключей:
1. Сначала сравниваются хэши ключей.
2. Если хэши равны и оба ключа реализуют Comparable, вызывается compareTo().
3. Если ключи не реализуют Comparable, используется tieBreakOrder(): сначала сравнение через названия классов, затем через System.identityHashCode().

Зачем нужен HashMap, если есть HashTable?
- Методы HashTable синхронизированы, что снижает производительность.
- HashTable не может содержать null, тогда как HashMap допускает один null-ключ и любое количество null-значений.
- Iterator у HashMap работает по принципу fail-fast, в отличие от Enumeration у HashTable.
- HashTable -- устаревший класс.

## Code
```java
import java.util.*;

public class HashMapInternals {
    public static void main(String[] args) {
        // Default: capacity=16, loadFactor=0.75
        Map<String, Integer> map = new HashMap<>();

        // Preallocate to avoid resizing
        // For 100 entries: 100 / 0.75 = 134, next power of 2 = 256
        Map<String, Integer> optimized = new HashMap<>(256);

        // How put() works internally:
        // 1. hash = spread(key.hashCode()) -- XOR upper/lower 16 bits
        // 2. index = (capacity - 1) & hash
        // 3. If bucket empty: insert new Node
        // 4. If bucket occupied: traverse chain
        //    - If key found (equals): replace value
        //    - If not found: append to chain
        // 5. If chain length >= 8 AND table >= 64: treeify
        // 6. If size > capacity * 0.75: resize (double)
        map.put("name", 1);
        map.put("name", 2); // replaces, returns old value 1

        // equals/hashCode contract demo
        Map<Person, String> people = new HashMap<>();
        Person p1 = new Person("Alice", 30);
        people.put(p1, "Engineer");
        System.out.println(people.get(new Person("Alice", 30))); // "Engineer"
        // Only works because Person overrides both equals AND hashCode
    }
}

class Person {
    private final String name;
    private final int age;

    Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Person)) return false;
        Person p = (Person) o;
        return age == p.age && Objects.equals(name, p.name);
    }

    @Override
    public int hashCode() {
        return Objects.hash(name, age); // MUST override with equals
    }
}
```

## Interview
### [6-2-q0 | junior] How does HashMap determine which bucket to place an entry in?
HashMap computes a hash by taking the key's hashCode() and XORing its upper 16 bits with its lower 16 bits (spread function) to reduce collisions. The bucket index is calculated as (capacity - 1) & hash, which works because capacity is always a power of 2. This is a fast bitwise AND that replaces the modulo operation.

### [6-2-q1 | mid] What happens when a HashMap bucket has too many collisions? What are the thresholds?
When a bucket's linked list reaches 8 nodes (TREEIFY_THRESHOLD), HashMap converts the list into a red-black tree, improving worst-case lookup from O(n) to O(log n). However, treeification only occurs if the table capacity is at least 64 (MIN_TREEIFY_CAPACITY); otherwise, it resizes the table instead. When removals shrink a tree below 6 nodes (UNTREEIFY_THRESHOLD), it converts back to a linked list. The gap between 8 and 6 prevents thrashing between list and tree forms.

### [6-2-q2 | senior] Explain the Java 8 resize optimization for HashMap. Why must capacity be a power of 2?
Power-of-2 capacity enables the bitwise AND trick: index = (capacity - 1) & hash, which is faster than modulo. During resize (doubling), each entry's new position depends on a single bit: the bit at the position of the old capacity. If that bit is 0, the entry stays at the same index; if 1, it moves to oldIndex + oldCapacity. This allows splitting each bucket into two using simple bit checks without recalculating hashes. Entries are split into 'lo' and 'hi' lists, maintaining order. This optimization, introduced in Java 8, also eliminated the possibility of infinite loops during concurrent resize that plagued Java 7's head-insertion rehashing.

## Checkpoint
### [6-2-cp0] Вы вставляете пользовательский объект как ключ, затем вызываете `get` с равным объектом — возвращается `null`. В чём наиболее вероятная причина?
Класс переопределяет `equals`, но не `hashCode`. Два логически равных объекта дают разные хэш-коды, поэтому попадают в разные бакеты. Вызов `get` ищет в неверном бакете и ничего не находит. Решение: всегда переопределяйте `hashCode` вместе с `equals`, чтобы равные объекты давали одинаковый бакет.

### [6-2-cp1] Связанный список в бакете достиг 8 узлов. При каком условии treeify НЕ происходит и что случается вместо этого?
Treeify пропускается, если capacity таблицы меньше MIN_TREEIFY_CAPACITY (64). Вместо преобразования в красно-чёрное дерево выполняется resize всей таблицы (удвоение), что перераспределяет записи по большему числу бакетов и уменьшает длину цепочки без накладных расходов на дерево.

## Key Terms
### load factor
Коэффициент (по умолчанию 0.75), определяющий момент resize в HashMap. Когда число записей превышает `capacity × load factor`, таблица удваивается. Меньший load factor снижает коллизии ценой большего расхода памяти; больший — экономит память, но увеличивает частоту коллизий.

### treeify
Процесс преобразования связанного списка в бакете HashMap в красно-чёрное дерево, когда список достигает TREEIFY_THRESHOLD (8 узлов) и таблица имеет не менее MIN_TREEIFY_CAPACITY (64) бакетов. Treeify улучшает худший случай поиска с O(n) до O(log n). Обратное преобразование (в список) происходит ниже UNTREEIFY_THRESHOLD (6 узлов).

### hash collision
Ситуация, при которой два различных ключа дают одинаковый индекс бакета после хэширования. HashMap разрешает коллизии через separate chaining (связанный список или красно-чёрное дерево в каждом бакете). Частые коллизии деградируют производительность и могут быть вызваны плохой реализацией `hashCode`, неравномерно распределяющей ключи.
